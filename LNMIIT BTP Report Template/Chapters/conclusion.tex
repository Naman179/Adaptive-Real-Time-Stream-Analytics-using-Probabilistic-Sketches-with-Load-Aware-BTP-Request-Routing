\section{Summary of Phase~1 Achievements}

Phase~1 of the \textit{Adaptive Real-Time Stream Analytics} project has successfully achieved all proposed foundational milestones:

\begin{enumerate}
    \item \textbf{Stream Ingestion:} A robust Kafka-based ingestion pipeline was built with support for configurable Zipfian traffic generation and concept drift simulation.
    \item \textbf{Probabilistic Sketches:} Three core algorithms --- Count-Min Sketch~\cite{cms}, HyperLogLog~\cite{hll}, and Sliding-Window Misra--Gries~\cite{misra} --- were implemented with mathematically verified error bounds.
    \item \textbf{Dual-Language Workers:} Both Python and high-performance C++ analytics workers were developed as interchangeable drop-in components writing to the same Redis state store.
    \item \textbf{Federated State Management:} Redis~\cite{redis} was deployed as a shared blackboard with TTL-based automatic eviction of stale worker entries.
    \item \textbf{REST API:} A comprehensive FastAPI~\cite{fastapi} backend was built with query, metrics, and control endpoints.
    \item \textbf{Real-Time Dashboard:} An interactive single-page web dashboard provides live visualization of all sketch estimates, system health, and worker performance metrics.
    \item \textbf{Testing:} A 28-test Python unit test suite and a standalone simulation benchmark validate correctness without requiring any external infrastructure.
\end{enumerate}

\section{Phase~2 Achievements (Completed)}

Building upon the Phase~1 foundation, we have now successfully implemented all proposed adaptive components:

\begin{enumerate}
    \item \textbf{Dynamic Parameter Controller:} A background asyncio service running every 5 seconds monitors per-worker P99 latency, memory pressure, and estimated sketch error. Based on configurable thresholds, the controller automatically switches workers between Low, Medium, and High precision templates without dropping events. The controller minimises the cost function:
    \begin{equation}
        C(\theta, r) = \alpha \cdot \text{LatencyViolation}(\theta, r) + \beta \cdot \text{Memory}(\theta) + \gamma \cdot \text{AccuracyPenalty}(\theta, r)
    \end{equation}
    where $\theta$ denotes the set of sketch parameters and $r$ is the routing decision.

    \item \textbf{Load-Aware Request Router:} Incoming queries are scored against all alive worker nodes using a multi-criteria utility function:
    \begin{equation}
        \text{score}_i(q) = w_1 \cdot \frac{A_i^{\text{capability}}}{\alpha_q} - w_2 \cdot \frac{L_i^{p95}}{L_q} - w_3 \cdot M_i^{\text{usage}}
    \end{equation}
    Queries are forwarded to the highest-scoring node, balancing accuracy capability, latency, and memory utilization.

    \item \textbf{Comparative Evaluation:} A systematic simulation benchmark comparing all four experimental configurations (static baseline, adaptive-only, router-only, fully adaptive) has been completed, demonstrating a 16\% improvement in P99 latency with adaptive parameter tuning.

    \item \textbf{Concept Drift Simulation:} The producer now includes a \texttt{--shift-interval} feature that rotates the Zipfian rank distribution at configurable intervals, demonstrating the sliding-window Misra--Gries sketch's ability to detect and adapt to non-stationary data distributions.
\end{enumerate}

\section{Future Work}

With the complete adaptive pipeline now operational, the following research extensions are planned:

\begin{itemize}
    \item \textbf{ML-Based Adaptive Controller:} Replacing the rule-based threshold controller with a Reinforcement Learning agent (Q-Learning or Deep Q-Networks). The RL policy $\pi$ would map the continuous system state (latency, memory, error) to optimal parameter actions, enabling self-tuning behaviour that adapts to specific hardware profiles and proactive control before thresholds are breached.

    \item \textbf{Bounding CMS Error Over Infinite Streams:} The current CMS accumulates counts indefinitely, causing the absolute error bound $\epsilon N$ to grow without limit. Two mitigations are planned:
    \begin{itemize}
        \item \textit{Conservative Updating:} Only incrementing the minimum counter for a given key, suppressing artificial inflation common in Zipfian distributions.
        \item \textit{Time-Decayed (Fading) Sketches:} Periodically scaling down all counters to exponentially decay older traffic, restricting $N$ to a recency-weighted sliding window.
    \end{itemize}

    \item \textbf{Sketch-Based Concept Drift Detection:} Implementing an automated similarity detection service that compares consecutive sketch windows using $L_2$ distance, enabling the controller to mathematically detect distribution shifts and trigger proactive sketch re-warming.

    \item \textbf{Hierarchical CMS for Range Queries:} Maintaining a tree of sketches at power-of-two (dyadic) granularities to enable range queries such as ``how many events appeared in $[T_1, T_2]$?'' in $O(\log N)$ time.

    \item \textbf{Distributed Controller Consensus:} Implementing a Raft or Paxos-based consensus group for the controller to eliminate the single point of failure in production deployments.

    \item \textbf{Hardware Acceleration (FPGA / P4):} Offloading sketch update logic to programmable switches or FPGA-based SmartNICs for per-packet processing at 10\,Gbps+ line rates.
\end{itemize}
